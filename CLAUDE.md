# AI Secretary 프로젝트 작업 원칙

## 범용성 원칙 (절대 준수)

코드·프롬프트·테스트·주석 작성 시 특정 사용자/지역/도메인/정치 성향을 하드코드하지 않는다.

사용자가 디버깅 중 특정 원고(예: 특정 인물명, 지역명, 정책 주제)를 예시로 제시해도 그건 **테스트용 표본**일 뿐이다. 오늘은 A이지만 내일은 B일 수 있고, 이 프로젝트는 모든 사용자에게 동일하게 동작해야 한다.

### 필수 규칙
- 실제 사용자명·실제 지역명을 코드·주석·프롬프트 템플릿·테스트 fixture 에 적지 말 것.
- 실패 재현 테스트를 만들 때도 placeholder 를 쓸 것: `홍길동`, `샘플구`, `샘플동`, 또는 `{name}`, `{region}` 같은 필드 표기.
- 주석으로 "예: 박지상 독립운동가 후손" 처럼 특정 사용자 상황을 예시로 적지 말 것. 일반화된 설명으로 대체 (예: "일반 주제에서는 1~2개만 뽑힐 수 있다").
- 프롬프트 few-shot 예시도 특정 사용자 메타데이터를 배제하고 구조만 드러내는 일반 예시로 작성.

### 예외
- 테스트 fixture 에서 `홍길동`, `아무개`, `샘플` 같이 **명백한 placeholder** 성격의 이름/지명은 허용.
- 실제 프로덕션 에러 로그를 조사할 때 로그 자체를 읽는 건 당연히 허용 — 단, 그 내용을 코드·주석·영구 저장 파일에 복제하지 말 것.

### 왜
- 이 시스템은 정치·행정 관련 메시지를 다양한 사용자에게 제공한다. 특정 인물의 메시지·성향·지역이 코드에 박히면 다른 사용자에 대한 편향으로 이어진다.
- 사용자가 반복적으로 지적한 원칙이다 ([memory/feedback_no_user_specific_hardcoding.md](../../../Users/강정구학부졸업미디어문예창작학과/.claude/projects/e--ai-secretary/memory/feedback_no_user_specific_hardcoding.md)).

## 병렬 세션 간섭 금지 규약 (절대 준수)

이 프로젝트는 여러 Claude 세션이 동시에 돌 수 있다. 너는 다른 세션의 작업을 망가뜨리지 않아야 한다. 다음 규칙을 모든 작업 내내 강제로 지켜라.

### 1. 작업 레인 선언
이 세션의 작업 범위는 사용자가 명시하기 전까지는 "사용자가 이번 대화에서 직접 지시한 작업과 그에 직결되는 파일뿐" 이다. 지시가 모호하면 작업 시작 전에 "이번 세션에서 건드릴 파일/디렉토리가 무엇인지" 사용자에게 먼저 확인하라. 레인 밖의 파일은 편집·이동·삭제 금지. 디버깅을 위한 참조 읽기는 허용하지만, 편집은 금지.

### 2. 커밋 스코프
`git add .` / `git add -A` / `git add -u` / `git commit -a` 금지. 반드시 `git add <파일 경로>` 로 이번 세션에서 직접 수정한 파일만 명시적으로 스테이징한다. `git status` 에 내가 만지지 않은 modified/untracked 가 보여도 무시하고 건드리지 마라 — 그건 다른 세션의 진행 중 작업이다.

### 3. 배포 전 게이팅
`firebase deploy` 는 git 을 보지 않고 디스크의 소스 디렉토리를 그대로 업로드한다. 배포하기 직전에 반드시 다음을 확인하라:
- (a) `git status` 에서 배포 대상 디렉토리(예: `functions/`) 가 clean 한가 — 수정·untracked 파일 0개
- (b) 로컬 브랜치 HEAD 가 `origin/<branch>` 와 동기화돼 있는가 (`git fetch && git status` 로 "up to date" 확인)

둘 중 하나라도 어긋나면 배포를 중단하고 사용자에게 "다른 세션의 미커밋/미푸시 작업이 보입니다. 어떻게 진행할까요?" 라고 물어라. 임의 판단으로 배포하지 마라.

### 4. 파괴적·전역 작업 금지
`git reset --hard`, `git checkout -- .`, `git clean`, `rm -rf`, `git push --force`, 마이그레이션 일괄 실행 같이 레포·프로덕션 전역에 영향 주는 작업은 사용자의 명시적·구체적 지시 없이는 절대 금지. 병렬 세션 환경에서는 한 번의 실수가 다른 세션의 수 시간 작업을 날린다.

### 5. 불확실하면 멈춤
어떤 행동이 다른 세션의 작업을 건드릴 가능성이 있다고 판단되면, 실행하지 말고 먼저 사용자에게 질문하라. 막힌 10초가 날아간 10시간보다 낫다.

### 왜
Firebase 는 단일 프로젝트고, 여러 세션이 같은 working tree 와 같은 배포 타깃을 공유한다. 세션 A 가 편집 중인 파일을 세션 B 가 커밋·배포하면 A 의 미완성 코드가 프로덕션에 올라가거나 A 의 변경이 사라진다. 이 규약은 그런 교차 오염을 막기 위한 것이다.

## H2 소제목 — deterministic fallback 템플릿 금지

H2 파이프라인에서 LLM repair가 실패했을 때 "{키워드} 왜 거론되나?", "{키워드} 경쟁력은?" 같은 기계적 템플릿으로 소제목을 생성하지 않는다. LLM이 만든 H2가 scoring을 통과하지 못하고 repair도 실패하면 **원본 H2를 유지**한다. 기계적 템플릿은 모든 글에서 동일한 패턴이 반복되어 콘텐츠 품질을 저하시킨다.
