# /clarify - 요구사항 명확화

소크라테스식 질문을 통해 모호한 아이디어를 구체적인 요구사항으로 정제합니다.

## 사용법

```
/clarify <아이디어>
/clarify 키워드 밀도 방식 변경
/clarify 새로운 기능 추가
```

## 핵심 원칙

**한 번에 하나의 질문만**. 사용자 답변을 기반으로 다음 질문 진행.

## 질문 영역 (5가지)

### 1. 목표 & 성공 기준
- "이 변경으로 무엇을 달성하고 싶으신가요?"
- "성공했다면 어떤 모습일까요?"
- "현재 방식의 문제점이 뭔가요?"

### 2. 범위 & 경계
- "어디까지 변경해야 하나요?"
- "변경하지 않을 부분은?"
- "1단계로 먼저 할 것과 나중에 할 것은?"

### 3. 컨텍스트 & 제약
- "관련된 기존 코드나 시스템이 있나요?"
- "기술적 제약이 있나요?"
- "시간/비용 제약은?"

### 4. 사용자 & 이해관계자
- "누가 이 기능을 사용하나요?"
- "관리자/테스터/일반 사용자 중 누구를 위한 건가요?"
- "사용자가 기대하는 동작은?"

### 5. 구체적 세부사항
- "예시를 들어주실 수 있나요?"
- "엣지 케이스는 어떻게 처리할까요?"
- "에러 상황은?"

## 진행 흐름

```
1. 아이디어 수신
2. 관련 코드 검색 (Glob/Grep으로 컨텍스트 파악)
3. 첫 질문 (보통 "목표 & 성공 기준"부터)
4. 답변 기반으로 다음 질문
5. 충분히 명확해지면 요약
6. 사용자 확인 후 종료
```

## 종료 조건

- 아이디어가 구체적이고 명확해졌을 때
- 사용자가 "됐어", "이제 구현해" 등으로 종료 신호를 보낼 때
- 5개 영역 중 3개 이상이 명확해졌을 때

## 최종 출력

```markdown
## 요구사항 정리

**목표**: [한 줄 요약]

**범위**:
- 포함: [...]
- 제외: [...]

**제약사항**: [...]

**구현 방향**: [...]
```
